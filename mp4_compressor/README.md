# mp4c — mp4を品質を確認しながら圧縮するCLI

mp4をHEVC(H.265)に再エンコードして容量を削減し、圧縮後に**VMAF**(知覚品質スコア)で
「見た目がどれだけ元と変わったか」を測定して報告するツール。

- 元ファイルは絶対に上書きしない(出力先が既存なら中断)
- 圧縮後に元より大きくなったら出力を削除して報告
- 音声とメタデータは無劣化コピー(`-c:a copy`)
- 出力には `hvc1` タグを付与(QuickTime / iPhoneでの再生に必須)

## 必要なもの

- ffmpeg / ffprobe(libvmaf有効ビルド。Homebrewの `brew install ffmpeg` でOK)
- Rustツールチェーン

```bash
cargo build --release
# バイナリは target/release/mp4c
```

## 使い方

```bash
mp4c input.mp4                      # 既定: libx265 crf=22 preset=slow、VMAF測定あり
mp4c input.mp4 --target-vmaf 95     # 目標品質を満たす最小サイズの設定を自動探索(推奨)
mp4c input.mp4 --fast --target-vmaf 95  # ハードウェアエンコードで自動探索(大きいファイル向き)
mp4c input.mp4 -o out.mp4           # 出力先を指定(既定: input.compressed.mp4)
mp4c input.mp4 --crf 24             # CRFを手動指定してもう少し攻めた圧縮
mp4c input.mp4 --preset medium      # エンコードを速く(圧縮効率は少し落ちる)
mp4c input.mp4 --fast               # ハードウェアエンコード(下記参照)
mp4c input.mp4 --fast --qv 70       # ハードウェアエンコードの品質指定
mp4c input.mp4 --no-vmaf            # VMAF測定をスキップ

# バッチ処理
mp4c a.mp4 b.mp4 c.mp4              # 複数ファイルを順に処理
mp4c videos/                        # ディレクトリ直下の*.mp4を一括処理
mp4c videos/ --out-dir compressed/  # 出力先ディレクトリを指定(元と同じファイル名)
```

バッチ処理の挙動:

- 1件の失敗(壊れたファイル等)では止まらず、次のファイルへ進んで最後にサマリーを表示する
  (1件でも失敗があれば終了コードは1)
- ディレクトリ展開時、過去の出力(`*.compressed.mp4`)は自動でスキップする
- 出力先が既に存在するファイルはエラー扱いで飛ばすので、途中で中断しても
  同じコマンドの再実行で残りだけ処理される
- `-o/--output` は入力が1ファイルのときのみ。複数入力には `--out-dir` を使う

## オプション

| オプション | 既定値 | 説明 |
|---|---|---|
| `-o, --output <PATH>` | `<input>.compressed.mp4` | 出力パス(入力1ファイル時のみ) |
| `--out-dir <DIR>` | — | 出力先ディレクトリ(無ければ作成、ファイル名は入力と同じ) |
| `--crf <N>` | 22 | x265のCRF。**小さいほど高品質・大サイズ**。目安 20〜26 |
| `--target-vmaf <N>` | — | 目標VMAF(推奨 93〜97)。品質設定を自動決定する。`--crf`/`--qv`/`--cq`と排他、`--fast`と併用可 |
| `--preset <NAME>` | slow | x265のpreset。`medium`で数倍速く、`slow`で数%小さく |
| `--fast` | off | ハードウェアエンコーダを使う(下記参照) |
| `--qv <0-100>` | 65 | `--fast`(VideoToolbox)時の品質。**大きいほど高品質・大サイズ**。目安 55〜75 |
| `--cq <0-51>` | 24 | `--fast`(NVENC)時の品質。**小さいほど高品質・大サイズ**。目安 22〜28 |
| `--no-vmaf` | off | エンコード後のVMAF測定をスキップ |

## ソフトウェア(既定) vs ハードウェア(`--fast`)

`--fast` は実行時に `ffmpeg -encoders` を見て、使えるハードウェアエンコーダを
**VideoToolbox(Mac) → NVENC(NVIDIA GPU)** の優先順で自動選択する。
どちらも無ければエラーになる。

|  | 既定 (libx265) | `--fast` |
|---|---|---|
| エンコード方式 | CPUによるソフトウェアエンコード | Apple Siliconのメディアエンジン / NVIDIA GPUのNVENC |
| 速度 | 遅い(実時間の数倍かかることも) | 数倍〜10倍以上速い、低発熱 |
| 圧縮効率 | 高い | 同じ見た目の品質でサイズが**2〜5割大きい** |
| 品質指定 | `--crf`(小さいほど高品質) | VideoToolbox: `--qv`(大きいほど高品質) / NVENC: `--cq`(小さいほど高品質) |

「容量を最小にしたい」なら既定のlibx265、「多少大きくてもいいから早く終わらせたい」なら
`--fast`、という使い分け。`--qv 65` / `--cq 24` が `--crf 22` に近い感覚だが、
動画の内容によってズレるのでVMAFの報告を見て調整する。

- x265はGPU/CUDAでは速くならない(ほぼ純CPU実装)。遅いときは `--preset medium` が効く
- VideoToolboxの品質指定(`-q:v`)はApple Silicon専用。IntelのMacではビットレート指定しか
  効かないため、期待した品質にならないことがある
- VMAF測定は常にCPUで行われる(全コア使用)

## Linux (CUDA / NVENC) での利用

Rust側の依存はffmpeg CLIだけなので、Linuxでもそのまま `cargo build --release` で動く。

- NVENCの利用にはNVIDIAドライバと、NVENC有効ビルドのffmpegが必要。
  Ubuntuのapt版ffmpegは有効なことが多い。確認: `ffmpeg -encoders | grep nvenc`
- CUDAがあってもlibx265とVMAF測定は速くならない(CPU実装)。
  VMAFのCUDA版(`libvmaf_cuda`)は存在するが、apt版ffmpegには入っていないため
  使うなら自前ビルドが必要
- Intel/AMD内蔵GPU(VAAPI)は未対応

## 自動品質探索 (`--target-vmaf`)

最適な品質設定は映像の内容(アニメ / 実写 / グレインの多さ)によって動画ごとに違う。
`--target-vmaf 95` を指定すると、その動画にとって「目標品質を満たす最も圧縮の効く
設定(=最小サイズ)」を自動で見つけてからフル尺エンコードする。

仕組み(ab-av1方式):

1. 動画の 1/6・3/6・5/6 の位置から各20秒のサンプル区間を切り出す(再エンコードなしで高速)
2. 候補設定でサンプルだけエンコードしてVMAFを測定。判定には**最悪サンプルの値**を使う
   (静かな区間だけ見て、激しい区間で品質が足りなくなるのを避けるため)
3. 品質パラメータを二分探索する。探索対象はエンコーダによって変わる:
   - libx265(既定): CRF 18〜30(試行は最大4回)
   - `--fast` VideoToolbox: qv 45〜85(試行は最大6回。1試行が速いので実時間はむしろ短い)
   - `--fast` NVENC: cq 20〜38(試行は最大5回)
4. 見つかった設定でフル尺を1回だけエンコードし、最後に全体のVMAFも測定して報告

補足:

- 2分以下の短い動画はサンプリングせず全体で探索する
- バッチ処理では**ファイルごとに**探索する(動画ごとに最適設定が違うため)
- 探索はフル尺と同じpresetで行うので、`--preset medium` にすると探索も本番も速くなる
- 最も高品質側の設定でも目標に届かない映像(グレインが極端に多い等)はエラーで知らせる
- 数GB級の大きいファイルは `--fast --target-vmaf 95` が現実的
  (品質固定の `--fast` 単体と違い、出来上がりの品質が保証される)

## VMAFスコアの読み方

元動画と圧縮後を比較した知覚品質スコア(0〜100)。

| スコア | 意味 |
|---|---|
| 95以上 | ほぼ見分けがつかない。もっと攻めた設定でさらに縮む余地あり |
| 90〜95 | 注視すれば違いが分かる程度。通常の視聴には十分 |
| 90未満 | 劣化が知覚できる可能性あり。`--crf` を下げる / `--qv` を上げる |

測定結果に応じて、次に試すオプションをツールが提案する。

なお、`testsrc2` などの人工映像はVMAFが極端に高く出るため、実写の動画で試すこと。
